## 1. 데이터베이스 스키마 변경

### 1.1 새로운 테이블: 학과진입요건 (Major Entry Requirements) - ✅ **구현 완료**
- **구현된 필드:**
  - ✅ department_id (학과 ID)
  - ✅ admission_year (입학년도)
  - ✅ requirement_group (요건 그룹)
  - ✅ target_grade_level (대상 학년, Enum)
  - ✅ required_count (필요 과목 수)
  - ✅ logic_operator (논리 연산자: AND/OR)
  - ✅ requirement_text (사용자 노출용 설명)
  - ✅ is_alert_required (학생설계전공 알림 여부)
  - ✅ course_code (과목코드) - RequirementCourse 테이블로 분리

- **특이사항:**
  - ✅ OR 조건: 같은 requirement_group에 여러 레코드 + logic_operator="OR"
  - ✅ UniqueConstraint(department_id, admission_year, requirement_group)
  - ✅ 학생설계전공 알림 기능 (is_alert_required)
  - ❌ **선택옵션 필드 불필요** - 현재 구조로 충분히 표현 가능

### 1.2 학생수강과목 테이블 수정 - ✅ **이미 완료**
- **현재 상태:**
  - ✅ 전공진입조건 관련 필드 없음 (별도 테이블로 분리됨)
  - ✅ 권장과목 여부 필드 없음 (별도 관리)
  - ✅ CourseEnrollment 테이블이 깨끗하게 설계됨
- **결론:** 요구사항에서 제거하려던 필드가 애초에 없음. 이미 올바르게 분리됨.

### 1.3 새로운 테이블: 학생별 전공 평가 결과 - ✅ **구현 완료** (StudentRequirementStatus)
- **구현된 필드:**
  - ✅ student_id (학생 ID)
  - ✅ department_id (학과 ID)
  - ✅ is_satisfied (충족 여부 Boolean)
  - ✅ analysis_json (상세 분석 데이터 JSON)
  - ✅ gpa_score (전체 학점 점수 0-100)
  - ✅ required_courses_score (전공진입필수과목 충족도 0-100)
  - ✅ recommended_completion_score (권장과목 이수도 0-100)
  - ✅ recommended_grade_score (권장과목 학점 점수 0-100)
  - ✅ curriculum_completion_score (교육과정 완성도 0-100)
  - ✅ overall_score (종합 점수 0-100, 가중 평균)
  - ✅ ai_summary (AI 총평 텍스트)
  - ✅ calculated_at (계산 시간)

- **특징:**
  - ✅ 모든 학과에 대해 평가 결과 저장 가능
  - ✅ 학생 1명당 여러 학과의 평가 데이터 존재
  - ✅ UniqueConstraint(student_id, department_id)로 중복 방지
  - ❌ **평가 알고리즘 미구현** - 테이블만 정의됨

## 2. 성능 최적화 전략 - ❌ 미구현

### 2.1 사전계산 (Pre-computation) 방식 채택
- **배경:**
  - 페이지 요청마다 20-30명씩 실시간 계산하는 것은 비효율적
  - 450명 전체 학생 데이터 기준으로 사전 계산 필요

- **구현 방식:**
  - 데이터 업데이트 시점에 모든 학생의 전공 진입 조건 체크 수행
  - 계산 결과를 DB 테이블에 저장
  - 프론트엔드는 저장된 결과만 조회하여 표시

### 2.2 캐싱 전략 (선택사항)
- **Option 1 - Lazy Loading:**
  - 초기에는 빈 테이블 상태
  - 페이지에서 정보 요청 시 계산 후 DB 저장
  - 이후 접속 시 저장된 데이터 사용

- **Option 2 - Full Pre-calculation:**
  - 데이터 업데이트마다 전체 학생 재계산
  - 로그 기록 유지 (변경 이력 추적)

## 3. 관리자 기능 - ❌ 미구현

### 3.1 데이터 업데이트 관리 프로그램
- **기능:**
  - 새로운 데이터 업로드 시 DB 테이블 자동 업데이트
  - 기존 테이블 갈아엎고 새로 생성 (또는 로그 남기기)
  
### 3.2 진단 결과 생성 모듈
- **요구사항:**
  - 수강 현황 기반 진단 문구 자동 생성
  - 실시간 처리 대신 사전 계산 후 저장
  - AI 총평 제외한 나머지 로직 먼저 구현

### 3.3 핵심 모듈 설계
- 데이터 업데이트 트리거 시 실행되는 배치 프로세스
- 조건 체크 로직:
  1. 테이블 검색
  2. 해당 학생 검색
  3. 학생 데이터 존재 시 가져오기
  4. 프론트엔드에 신호 전달

## 4. 프론트엔드 동작 방식 - ⚠️ 부분 구현

### 4.1 데이터 표시
- ✅ **구현됨:** 커리큘럼에서 진입요건 과목 "수강필수" 배지 표시
- ❌ **미구현:** DB에서 사전 계산된 평가 결과 조회 및 표시
- ❌ **미구현:** 디폴트로 1지망 학과 데이터 우선 표시

### 4.2 학과 선택 시
- ✅ **구현됨:** 커리큘럼 뷰에서 학과 선택 가능
- ❌ **미구현:** 전공진입조건 평가 문구 표시
- **두 가지 처리 방식 검토:**
  - Option 1: 백엔드에서 실시간 계산 (로딩 화면 필요)
  - Option 2: 사전 계산된 데이터 조회 (권장)

## 5. 알고리즘 개발 필요 사항 - ❌ 미구현

### 5.1 희망전공진입 평가 알고리즘 (핵심 난이도)
- **계산 항목:**
  - 전공진입필수과목 충족도
  - 권장과목 이수도
  - 전체적합도

- **고려사항:**
  - 학번마다 조건이 다름
  - 복잡한 조건 로직 필요

### 5.2 LLM 활용 검토
- 복잡한 정보를 DB에 넣지 않고 LLM에 질의응답
- AI 커멘트: 텍스트 요약 형태로 제공

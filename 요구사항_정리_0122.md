# 요구사항 정리 (2025-01-22 회의록)

## 1. 데이터베이스 스키마 변경

### 1.1 새로운 테이블: 학과진입요건 (Major Entry Requirements)
- **필드 구조:**
  - 학과명 (major_name)
  - 과목코드 (course_code)
  - 적용년도/입학년도 (applicable_year)
  - 유형 (type): 필수(required) / 권장(recommended)
  - 선택옵션 필드 (option_a, option_b, option_c): Boolean 타입
    - 예시: 과목 A, B, C 중 하나 선택 → A 수강시 `1,0,0`, B 수강시 `0,1,0`
    - 여러 과목 중 선택 가능한 경우를 표현하기 위한 플래그

- **특이사항:**
  - 학생설계전공은 기본값/처리방식을 다르게 적용
  - Alert 또는 알림창으로 사용자에게 안내

### 1.2 학생수강과목 테이블 수정
- **제거할 필드:**
  - 전공진입조건 관련 필드
  - 권장과목 여부 필드
- **이유:** 전공별, 년도별로 조건이 달라지므로 별도 테이블로 분리

### 1.3 새로운 테이블: 학생별 전공 평가 결과
- **필드:**
  - 학생 ID (student_id)
  - 학과명 (major_name)
  - 이수현황 (completion_status)
  - 수강과목적합성 (course_suitability)
  - 전공진입필수과목 충족도 (required_courses_status)
  - 권장과목 이수도 (recommended_courses_status)
  - 전체적합도 (overall_suitability)
  - AI 총평 (ai_summary)
- **특징:**
  - 모든 학과에 대해 평가 결과 저장
  - 학생 1명당 여러 학과의 평가 데이터 존재

## 2. 성능 최적화 전략

### 2.1 사전계산 (Pre-computation) 방식 채택
- **배경:**
  - 페이지 요청마다 20-30명씩 실시간 계산하는 것은 비효율적
  - 450명 전체 학생 데이터 기준으로 사전 계산 필요

- **구현 방식:**
  - 데이터 업데이트 시점에 모든 학생의 전공 진입 조건 체크 수행
  - 계산 결과를 DB 테이블에 저장
  - 프론트엔드는 저장된 결과만 조회하여 표시

### 2.2 캐싱 전략 (선택사항)
- **Option 1 - Lazy Loading:**
  - 초기에는 빈 테이블 상태
  - 페이지에서 정보 요청 시 계산 후 DB 저장
  - 이후 접속 시 저장된 데이터 사용

- **Option 2 - Full Pre-calculation:**
  - 데이터 업데이트마다 전체 학생 재계산
  - 로그 기록 유지 (변경 이력 추적)

## 3. 관리자 기능

### 3.1 데이터 업데이트 관리 프로그램
- **기능:**
  - 새로운 데이터 업로드 시 DB 테이블 자동 업데이트
  - 기존 테이블 갈아엎고 새로 생성 (또는 로그 남기기)
  
### 3.2 진단 결과 생성 모듈
- **요구사항:**
  - 수강 현황 기반 진단 문구 자동 생성
  - 실시간 처리 대신 사전 계산 후 저장
  - AI 총평 제외한 나머지 로직 먼저 구현

### 3.3 핵심 모듈 설계
- 데이터 업데이트 트리거 시 실행되는 배치 프로세스
- 조건 체크 로직:
  1. 테이블 검색
  2. 해당 학생 검색
  3. 학생 데이터 존재 시 가져오기
  4. 프론트엔드에 신호 전달

## 4. 프론트엔드 동작 방식

### 4.1 데이터 표시
- DB에서 사전 계산된 결과만 조회하여 표시
- 디폴트: 1지망 학과 데이터 우선 표시

### 4.2 학과 선택 시
- 사용자가 학과 선택 시 해당 학과의 전공진입조건 문구 표시
- **두 가지 처리 방식 검토:**
  - Option 1: 백엔드에서 실시간 계산 (로딩 화면 필요)
  - Option 2: 사전 계산된 데이터 조회 (권장)

## 5. 알고리즘 개발 필요 사항

### 5.1 희망전공진입 평가 알고리즘 (핵심 난이도)
- **계산 항목:**
  - 전공진입필수과목 충족도
  - 권장과목 이수도
  - 전체적합도

- **고려사항:**
  - 학번마다 조건이 다름
  - 복잡한 조건 로직 필요

### 5.2 LLM 활용 검토
- 복잡한 정보를 DB에 넣지 않고 LLM에 질의응답
- AI 커멘트: 텍스트 요약 형태로 제공

## 6. 개발 단계별 계획

### Phase 1: 기본 로직 구현
- AI 파트는 비워두고 나머지 로직 먼저 구현
- 현재 데이터로 알고리즘 개발 및 테스트

### Phase 2: 샘플 데이터 준비
- **필수:**
  - DB에 샘플 데이터 삽입
  - 새 테이블(전공진입필수, 권장과목, 전체적합도)은 초기 상태 빈 테이블

### Phase 3: AI 총평 연동
- AI 로그만 캐싱으로 사전 저장
- 전공진입 필수/권장과목/전체적합도 계산 완료 후 AI 통합

## 7. 기술적 결정 사항

### 7.1 데이터 처리 방식
- ✅ 사전 계산 방식 채택 (실시간 계산 X)
- ✅ 변경 로그 기록 유지
- ✅ 배치 업데이트 프로세스 구현

### 7.2 아키텍처
- 백엔드: 데이터 계산 및 저장 담당
- 프론트엔드: 데이터 조회 및 표시만 담당
- 관리자 모듈: 별도 구현

## 8. 우선순위 및 난이도

### 높은 난이도
- ⚠️ 희망전공진입 평가 알고리즘
- ⚠️ 학번별 조건 처리 로직

### 중간 난이도
- 학과진입요건 테이블 설계
- 사전계산 배치 프로세스

### 낮은 난이도
- 프론트엔드 데이터 조회 로직
- 샘플 데이터 준비

## 9. 추가 고려사항

- 학생설계전공: 일반 학과와 다른 처리 방식/알림 필요
- 전공진입조건 문구 자동 생성 기능
- 로딩 중 화면 UI 준비 (필요시)
- 데이터 업데이트 주기 및 트리거 정의 필요
